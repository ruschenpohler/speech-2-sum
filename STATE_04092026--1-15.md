# Project State Document — April 9, 2026

## 1. Project Overview

**Speech-2-Sum** is a local, private speech-to-text and summarization system that runs entirely on CPU — no cloud, no GPU required.

**Two transcription backends:**
| Model | Parameters | ToolKit | Best For |
|-------|------------|---------|----------|
| Voxtral Mini 4B | 4B | HuggingFace Transformers | Higher accuracy, streaming output |
| Parakeet TDT 0.6B | 0.6B | NVIDIA NeMo | Faster on CPU, batch-only |

**One summarization backend:**
- Qwen2.5-7B-Instruct (GGUF Q8_0) via llama.cpp
- Runs on CPU, ~12GB RAM required
- Produces abstractive summary + extractive key quotes

## 2. What We've Built

### Core Scripts
- `transcribe.py` — Dispatcher: `--voxtral` or `--parakeet`, optionally `--sum` for auto-summarization
- `transcribe_voxtral.py` — Voxtral transcription (streaming)
- `transcribe_parakeet.py` — Parakeet transcription (batch)
- `summarize.py` — Qwen summarization with truncation strategies (`--keep first/last`)
- `download_*.py` — Model downloaders for each backend
- `utils.py` — Shared utilities (RAM monitoring, WiFi/Location, transcript saving)

### Recording Features
- `--mic` — Record from physical microphone
- `--mic-loop` — Record system audio (WASAPI loopback) — captures whatever is playing through speakers/headphones
- `--mic --mic-loop` — Record both simultaneously, normalized and mixed

### Installation
- `install.py` — Interactive setup (creates venv, installs extras)
- `pyproject.toml` — Package config with optional dependencies: `voxtral`, `parakeet`, `summarize`, `all`

## 3. Current State

### Where We Are
We are in the middle of debugging and extending the `--mic-loop` (system audio recording) feature.

**Goal:** Enable users to capture meetings where:
- Their voice comes through the microphone (`--mic`)
- Other participants' voices come through system audio (`--mic-loop` — e.g., Zoom/Teams call playing through laptop speakers)
- Both can be recorded simultaneously and mixed

### What Works
- `--mic` alone — Records physical microphone correctly
- `--mic-loop` on systems with proper WASAPI loopback (e.g., external headphones, some USB audio interfaces)
- `--mic --mic-loop` — Recording both when both devices are functional

### What Doesn't Work (On This Machine)
- Built-in laptop speakers (Realtek Audio) don't produce a usable system audio capture device

## 4. Problems Encountered

### Problem 1: WASAPI Loopback Not Found

**Symptom:** When running `--mic-loop`, the system reports:
```
Warning: WASAPI loopback device not found — falling back to mic.
```

**Root Cause:** On Windows, built-in speaker output doesn't automatically create a WASAPI loopback device. The detection logic looks for devices that match the default output device name, but built-in speakers don't expose this.

**Solutions Tried:**
1. Added 30-second wait period to allow user to start playing audio (hoping the device would appear)
2. Added fallback to `PC Speaker` and `Stereo Mix` via WDM-KS API (these devices appear but don't capture audio)
3. Enabled `Stereo Mix` in Windows settings (device now appears via WASAPI at index 12)

**Current Status:** Device is detected, but opening it fails with:
```
PortAudioError: Error opening InputStream: Invalid sample rate [PaErrorCode -9997]
```

### Problem 2: Sample Rate Mismatch

**Symptom:** Even after detecting the Stereo Mix device (index 12, WASAPI), opening the stream fails with "Invalid sample rate".

**Analysis:** The code requests 16kHz (Parakeet's expected sample rate), but the device likely supports a different rate (e.g., 44100Hz or 48000Hz). We added code to query the device's `default_sample_rate` and adjust, but it still fails — indicating either:
- The device info returns an unsupported rate, or
- The device doesn't actually support the rate it claims

## 6. Where We Want to Go

### Immediate Goal
Make `--mic-loop` work on this laptop with built-in speakers. This involves:
1. Understanding why the WASAPI Stereo Mix device (index 12) fails to open
2. Finding the correct sample rate to use, or determining if the device is simply incompatible
3. Either fixing the device handling, or documenting that external audio (headphones/USB) is required

### Broader Goals
1. **Robust loopback detection** — Handle various Windows audio configurations (built-in speakers, HDMI, USB, Bluetooth)
2. **Mixed recording** — Ensure `--mic --mic-loop` works smoothly when both devices are functional
3. **Cross-platform** — Ensure the same logic works on Linux/macOS (where loopback typically works via PulseAudio/ALSA)
4. **Performance** — Optimize recording latency, handle clock drift between mic and loopback streams

### Technical Details for Future Work

**Device Detection Logic** (`utils.py:get_loopback_device`):
- First tries WASAPI loopback (matches default output device name or "loopback" in name)
- Falls back to WDM-KS devices: "PC Speaker" and "Stereo Mix"
- Filters by API type via `api_filter` parameter ("WASAPI" or "WDM-KS")

**Recording Logic** (`utils.py:record_audio`):
- Accepts `mic` and `loopback` booleans
- Opens 1 or 2 `sounddevice.InputStream` concurrently via `MultiStream` context manager
- Each stream uses a callback to append audio chunks to a list
- At end: normalizes both streams (to max amplitude 1.0) and averages them if both present
- Returns `(audio_array, source_label)`

**Sample Rate Handling:**
- Parakeet expects exactly 16kHz
- Voxtral uses `processor.feature_extractor.sampling_rate` (~16kHz)
- Current code attempts to adjust to device's `default_sample_rate`, but this may not be supported
